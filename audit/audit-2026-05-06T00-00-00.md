## CRITICAL

### C-01 — `console.log` Leaks Payment Data in Production
**Files:** `src/modules/payment/payment.controller.ts:184, 197`

`paymentFailController` and `paymentCancelController` unconditionally call `console.log` with the full SSLCommerz POST body, which includes transaction IDs, amounts, card info, and bank references. This runs in production.

```ts
// paymentFailController line 184
console.log("Payment failed query data:", data);
// paymentCancelController line 197
console.log("Payment cancelled query data:", data);
```

**Fix:** Remove both `console.log` calls. Production logging (if needed) should go through a structured logger and must strip sensitive fields.

### C-02 — Account Lock Off-by-One: Locks After 6 Failures, Not 5
**File:** `src/modules/auth/auth.service.ts:220-225`

`incrementFailedLoginCount` receives the *current* count and only sets `lockedAt` when `failedLoginCount >= 5`. This means locking triggers on the **6th** bad password (when the stored count is already 5), not the 5th.

```ts
data: {
  failedLoginCount: failedLoginCount + 1,
  ...(failedLoginCount >= 5 && { lockedAt: new Date() }),  // fires when count is already 5
}
```

Timeline:
- Attempts 1–5: stored count 0→5, no lock (condition `4 >= 5` is false on attempt 5)
- Attempt 6: stored count 5→6, **lock finally set** (`5 >= 5` is true)

AGENTS.md documents "locks after 5 failed attempts" — the code contradicts this.

**Fix:** Change to `failedLoginCount + 1 >= 5` (or equivalently `failedLoginCount >= 4`):
```ts
...(failedLoginCount + 1 >= 5 && { lockedAt: new Date() }),
```

## HIGH

### H-01 — Race Condition in Application Approval (TOCTOU)
**File:** `src/modules/application/application.service.ts:369-408`

The guard check for an already-approved application happens **outside** the database transaction. Two concurrent requests can both pass the check before either transaction commits, resulting in two APPROVED applications for the same task.

```ts
// OUTSIDE transaction — both concurrent requests can pass this check
const alreadyApprovedApplication = await prisma.application.findFirst({
  where: { taskId: application.taskId, status: "APPROVED" },
});
if (alreadyApprovedApplication) {
  throw new AppError(400, "This task already has an approved application");
}

// Transaction starts here — but the re-check is missing
await prisma.$transaction(async (tx) => {
  updatedApplication = await tx.application.update(...); // approve
  await tx.task.update({ data: { approvedApplicationId: applicationId, ... } });
});
```

**Impact:** Two applicants become APPROVED; `task.approvedApplicationId` / `task.agreedCompensation` gets set to whichever transaction commits last; double-payment is possible.

**Fix:** Move the `alreadyApprovedApplication` check inside the `$transaction` block so it runs within the same DB transaction and sees a consistent snapshot. Even better, add a partial unique index at the DB level: `UNIQUE (taskId) WHERE status = 'APPROVED'`.

### H-02 — Auth Middleware Returns 404 for Deleted Users Instead of 401
**Files:** `src/middlewares/authMiddleware.ts:42-44`, `src/modules/auth/auth.service.ts:160-164`

`getAuthUser` throws `AppError(404, "User not found")` when the user is missing or soft-deleted. Because this throw propagates through `asyncHandler`, clients with a valid JWT belonging to a deleted account receive **404**, not 401. This leaks the distinction between "token is invalid" and "token is valid but account was deleted".

```ts
// auth.service.ts
if (!user) {
  throw new AppError(404, "User not found", "NOT_FOUND");  // 404 leaks info
}
```

**Fix:** Throw `AppError(401, "You are not authorized")` in `getAuthUser`, matching the other auth failure messages. The middleware's own `!user || user.isDeleted` check at line 43 is also now dead code (never reached because `getAuthUser` throws first) and can be removed.

### H-03 — No Rate Limiting on `/refresh-token` Endpoint
**File:** `src/app.ts:45-46` (existing limiters), `src/modules/auth/auth.route.ts:28`

Only `/login` and `/forgot-password` are rate-limited. The refresh-token endpoint is unprotected at the HTTP layer and can be hammered to exhaust DB connections or to search for valid refresh tokens. The email verification send endpoint also has only an in-service 1-minute cooldown with no HTTP-level rate limit.

**Fix:** Add a rate limiter for `/refresh-token` (e.g., 30 req/15 min) and `/send-verification-email` (e.g., 5 req/hour) in `app.ts`:
```ts
const refreshTokenLimiter = rateLimit({ windowMs: 15 * 60 * 1000, max: 30, ... });
app.use("/api/v1/auth/refresh-token", refreshTokenLimiter);
```

### H-04 — `deleteTaskService` Omits `deletedAt` and `deletedBy` on Soft Delete
**File:** `src/modules/task/task.service.ts:208-220`

`deleteTaskService` only sets `isDeleted: true` but omits `deletedAt` and `deletedBy`. The `deleteAccountService` in `user.service.ts:263-269` correctly sets all three fields. The Task schema has both `deletedAt` and `deletedBy` fields (per AGENTS.md), making their absence an inconsistency that breaks audit trail.

```ts
// task.service.ts — missing deletedAt and deletedBy
await prisma.task.update({
  where: { id: taskId },
  data: { isDeleted: true },  // should also set deletedAt and deletedBy
});
```

**Fix:**
```ts
data: { isDeleted: true, deletedAt: new Date(), deletedBy: userId },
```

### H-05 — Wallet Balance Compared as `Decimal >= number` Without Explicit Cast
**File:** `src/modules/payment/payment.service.ts:197`

`wallet.balance` is `Prisma.Decimal`. `commissionAmount` is a plain `number`. The comparison relies on JavaScript's implicit coercion of `Decimal` via its `valueOf()`. This works at runtime but is type-unsafe, inconsistent with how the wallet service handles the same comparison (`Number(balance) < Number(amount)` at `wallet.service.ts:183`), and fragile across Prisma version changes.

```ts
if (wallet.balance >= commissionAmount) {  // Decimal vs number — implicit coercion
```

**Fix:** Make the cast explicit, matching the wallet service's pattern:
```ts
if (Number(wallet.balance) >= commissionAmount) {
```

## MEDIUM

**File:** `src/modules/auth/auth.controller.ts:208, 227`

CLAUDE.md mandates `ResponseHandler` static methods and explicitly forbids the legacy `sendResponse()`. Two controllers still use the old helper:

- `sendVerificationController` (line 208): `sendResponse(res, 200, "Verification email sent successfully", null)`
- `verifyEmailController` (line 227): `sendResponse(res, 200, "Email verified successfully", { user, accessToken, refreshToken })`

**Fix:**
```ts
// sendVerificationController
return ResponseHandler.ok(res, "Verification email sent successfully", null, { path: req.path });

// verifyEmailController
return ResponseHandler.ok(res, "Email verified successfully", { user, accessToken, refreshToken }, { path: req.path });
```

### M-02 — No Request Validation on `/refresh-token` Route
**File:** `src/modules/auth/auth.route.ts:28`

```ts
router.post("/refresh-token", refreshTokenController);  // no validateRequest
```

All other state-changing auth routes have a `validateRequest(schema)` middleware. The refresh token is extracted directly from `req.body.refreshToken || req.cookies.refreshToken` with no input schema. A Zod schema checking that `body.refreshToken` is an optional string would enforce the contract.

**Fix:** Add a `refreshTokenValidationSchema` and wire it into the route.

### M-03 — `approveApplicationService` Result May Be `undefined` at Runtime
**File:** `src/modules/application/application.service.ts:380-413`

`updatedApplication` is declared with `let` outside the `$transaction` closure and assigned inside it. TypeScript sees this as potentially `undefined` after the transaction (line 411 adds a guard, but the flow already returned the value before reaching that guard). More importantly, if the transaction callback doesn't run (e.g., Prisma throws before assignment), `updatedApplication` stays `undefined` and the guard at line 410-412 throws a 500 without any context. The return type is also lost since `updatedApplication` is typed as `any`.

**Fix:** Return the value from the transaction directly:
```ts
const updatedApplication = await prisma.$transaction(async (tx) => {
  const app = await tx.application.update(...);
  // ...
  return app;
});
return updatedApplication;
```

### M-04 — Phone Validation Regex Is US-Only; Platform Is Bangladesh-Based
**File:** `src/modules/user/user.validator.ts:4`

```ts
const usPhoneRegex = /^\+?1?[-.\s]?\(?\d{3}\)?[-.\s]?\d{3}[-.\s]?\d{4}$/;
```

The platform uses BDT currency, SSLCommerz (Bangladesh gateway), and sends payment data with `cus_country: "Bangladesh"`. US phone format validation will reject all Bangladeshi numbers (e.g., `+8801XXXXXXXXX`). Users cannot complete their profiles if they have valid Bangladeshi numbers.

**Fix:** Replace the regex with a Bangladeshi phone pattern:
```ts
const bdPhoneRegex = /^\+?880?1[3-9]\d{8}$/;
```
Or use a more permissive international format if multi-country support is intended.

## LOW

### L-01 — `parseExpiryToMs` Silently Falls Back to 7 Days for Unknown Units
**File:** `src/modules/auth/auth.controller.ts:10-19`

If `JWT_EXPIRES_IN` or `JWT_REFRESH_EXPIRES_IN` is set to an unrecognized suffix (e.g., `"1w"`, `"1M"`), `parseExpiryToMs` returns 7 days with no error or warning.

```ts
default: return 7 * 24 * 60 * 60 * 1000;  // silent fallback
```

**Fix:** Throw or log an error for unrecognized units; or validate accepted formats in `src/config/index.ts` at startup alongside the existing required-variable checks.

### L-02 — Rate Limiter Threshold (20) Exceeds Account Lock Threshold (6)
**File:** `src/app.ts:29-35`

The `loginLimiter` allows 20 attempts per 15 minutes. Given C-02 above (lock triggers at 6 failures), 14 attempts remain "within rate limit" after the account is already locked. Once C-02 is fixed (lock at 5), this gap narrows but still allows 15 post-lock requests. The limiter offers little marginal protection. Consider lowering to 10 or aligning with the lockout threshold (5–6).

### I-02 — IPN Endpoint Has No IP Allowlist
**File:** `src/modules/payment/payment.route.ts:49`

`POST /online/ipn-validate/` is publicly accessible with no IP allowlisting for SSLCommerz servers. The service does re-validate every claim against the SSLCommerz API (`validateOnlinePaymentService` calls the validation URL at line 566), which is the correct defense. But without IP filtering, the endpoint is a target for enumeration and DoS. Consider adding an IP allowlist middleware for SSLCommerz's published IP ranges.

### I-03 — SSLCommerz Store Credentials Sent in GET Query String
**File:** `src/modules/payment/payment.service.ts:566`

```ts
url: `${config.sslcommerz.validationApiUrl}?val_id=${payload.val_id}&store_id=${...}&store_passwd=${...}&format=json`,
```

`store_passwd` is appended to a GET URL, which means it appears in server access logs, browser history, and HTTP referer headers. While this may be required by SSLCommerz's API contract, it's worth noting. If SSLCommerz offers a POST-based validation endpoint, prefer that.
